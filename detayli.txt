1.1. PROJE TANIMI VE AMACI
───────────────────────────────────────────────────────────────────────────────

Multi-Agent Orchestration System (MAOS), birden fazla yapay zeka ajanının
koordineli bir şekilde çalışarak yazılım projeleri ürettiği otonom bir
sistemdir.

Temel Özellikler:
  • 8 farklı uzman ajan (Planner, Researcher, Coder, Critic, Executor, vb.)
  • Dual Provider mimari (Groq + OpenRouter)
  • Phased execution (fazlı çalıştırma)
  • Gerçek kod çalıştırma (subprocess)
  • Web arayüzü (FastAPI + SSE)
  • Telegram bot entegrasyonu
  • Memory agent (proje hafızası)
  • Git entegrasyonu

Kullanım Senaryosu:
  Kullanıcı: "Python ile hesap makinesi yaz"
  Sistem:
    1. Planner hedefi analiz eder, görevlere böler
    2. Researcher internette araştırma yapar
    3. Coder Python kodu yazar
    4. Critic kodu puanlar
    5. Executor kodu çalıştırır, test eder
    6. Sonuç: Çalışır durumda bir proje

1.2. PROJE VİZYONU
───────────────────────────────────────────────────────────────────────────────

Kısa Vadeli Vizyon (2026):
  • Cursor ve Windsurf benzeri otonom kodlama asistanı
  • Mevcut projeleri düzenleme (/open komutu)
  • Cluster mode (paralel ajan çalıştırma)
  • Human-in-the-loop (insan onayı)

Orta Vadeli Vizyon (2027):
  • RAG (Retrieval-Augmented Generation) entegrasyonu
  • Vektör veritabanı (embeddings)
  • Çoklu dil desteği (JavaScript, TypeScript, Go, Rust)
  • SaaS platformu (aylık abonelik)

Uzun Vadeli Vizyon (2028+):
  • Swarm intelligence (sürü zekası)
  • Takım bazlı çalışma (5 coder, 5 planner)
  • Otonom DevOps (CI/CD entegrasyonu)
  • Blockchain entegrasyonu (akıllı sözleşme üretimi)

2.2. AJAN YAPISI (8 AJAN)
───────────────────────────────────────────────────────────────────────────────

Her ajan BaseAgent sınıfından türer ve şu özelliklere sahiptir:

BaseAgent Yapısı:
  • agent_id: Benzersiz kimlik
  • _llm: LLM client (model bağlantısı)
  • short_term_memory: Kısa süreli bellek
  • long_term_memory: Uzun süreli bellek
  • tools: Kullanabileceği araçlar

Ajan Yaşam Döngüsü:
  1. think() → Düşünme (analiz)
  2. act() → Eylem (araç kullanımı)
  3. observe() → Gözlem (sonuç analizi)
  4. iterate() → Tekrar (gerekirse)

Ajan İletişimi:
  • Message Bus üzerinden mesajlaşma
  • Context sharing (bağlam paylaşımı)
  • Tool results (araç sonuçları)

2.3. ORKESTRATÖR (ORCHESTRATOR)
───────────────────────────────────────────────────────────────────────────────

Orchestrator, tüm sistemi yöneten ana kontrol merkezidir.

Görevleri:
  1. Görev Yönetimi:
     • Görevleri fazlara ayırma
     • Görev sıralaması
     • Paralel görev çalıştırma
     • Görev bağımlılıkları

  2. Ajan Koordinasyonu:
     • Hangi ajanın ne zaman çalışacağı
     • Ajan arası mesajlaşma
     • Context paylaşımı

  3. Hata Yönetimi:
     • Retry mekanizması (3 deneme)
     • Rate limit handling
     • Timeout yönetimi
     • Graceful degradation

  4. Performans İzleme:
     • Token kullanımı
     • Maliyet hesaplama
     • Süre ölçümü
     • Başarı oranı

Phased Execution:
  Orchestrator görevleri fazlara ayırır:
  
  Faz 1: Veritabanı Katmanı
    • Veritabanı şeması
    • Model tanımları
    • Migration scriptleri
  
  Faz 2: İş Mantığı Katmanı
    • API endpoint'leri
    • Business logic
    • Servis katmanı
  
  Faz 3: Arayüz Katmanı
    • UI bileşenleri
    • Routing
    • State management

2.4. MESSAGE BUS (MESAJLAŞMA SİSTEMİ)
───────────────────────────────────────────────────────────────────────────────

Message Bus, ajanlar arası iletişimi sağlar.

Özellikler:
  • Asenkron mesajlaşma
  • Pub/Sub pattern
  • Message queue
  • Event broadcasting

Mesaj Tipleri:
  1. Task Messages:
     • task_created
     • task_started
     • task_completed
     • task_failed

  2. Agent Messages:
     • agent_thinking
     • agent_acting
     • agent_observing

  3. System Messages:
     • system_error
     • system_warning
     • system_info

2.5. LLM CLIENT (DUAL PROVIDER)
───────────────────────────────────────────────────────────────────────────────

LLM Client, iki farklı API sağlayıcısını destekler:

Provider 1: OpenRouter
  • Base URL: https://openrouter.ai/api/v1
  • Modeller:
    - openai/gpt-oss-120b ($0.039/$0.19)
    - openai/gpt-4o-mini ($0.15/$0.60)
    - qwen/qwen-2.5-coder-32b-instruct ($0.20/$0.20)
  • Özellikler:
    - 375+ model seçeneği
    - Tek API key
    - Otomatik fallback

Provider 2: Groq
  • Base URL: https://api.groq.com/openai/v1
  • Modeller:
    - llama-3.3-70b-versatile (BEDAVA!)
  • Özellikler:
    - Ultra hızlı (LPU çipleri)
    - Sınırsız bedava istek
    - Düşük latency

Model Routing:
  config/settings.py içinde tanımlı:
  
  MODEL_ROUTING = {
      "planner":    {"model": QWEN3_235B,   "provider": "openrouter"},
      "researcher": {"model": GPT_OSS_120B, "provider": "openrouter"},
      "coder":      {"model": QWEN_CODER,   "provider": "openrouter"},
      "critic":     {"model": GPT_OSS_120B, "provider": "openrouter"},
      "executor":   {"model": GPT_OSS_120B, "provider": "openrouter"},
  }

Rate Limit Handling:
  • Exponential backoff (30-60-120 saniye)
  • Otomatik retry (3 deneme)
  • Graceful degradation

3.4. CRITIC AGENT (ELEŞTİRMEN)
───────────────────────────────────────────────────────────────────────────────

Dosya: agents/critic_agent.py
Model: openai/gpt-oss-120b (OpenRouter)
Maliyet: ~$0.0005/görev

Görev:
  Kodu 5 kritere göre puanlar, kalite kontrol yapar.

Puanlama Kriterleri:
  1. Correctness (Doğruluk): 0-10
     • Kod çalışıyor mu?
     • Hata var mı?
     • Mantık doğru mu?
  
  2. Quality (Kalite): 0-10
     • PEP8 uyumlu mu?
     • Type hints var mı?
     • Docstring var mı?
  
  3. Test Coverage (Test Kapsamı): 0-10
     • Test var mı?
     • Edge case'ler test edilmiş mi?
  
  4. Architecture (Mimari): 0-10
     • SOLID prensipleri
     • Design patterns
     • Modülerlik
  
  5. Security (Güvenlik): 0-10
     • SQL injection
     • XSS
     • Input validation

Karar Mekanizması:
  • Ortalama ≥ 7.0 → ONAYLA (approve)
  • Ortalama 4.0-6.9 → REVİZYON (revise)
  • Ortalama < 4.0 → YENİDEN YAZ (reject)

Çıktı Formatı (JSON):
  {
    "scores": {
      "correctness": 8,
      "quality": 7,
      "test_coverage": 6,
      "architecture": 8,

Fonksiyon: _extract_file_blocks()
Tarih: 02.03.2026

Sorun:
  • Qwen2.5-Coder [FILE:] formatını kullanmıyordu
  • Kendi formatını kullanıyordu: ```python # name.py
  • Parse edilemiyordu
  • Fallback kullanılıyordu

Çözüm:
  • 3 format desteği eklendi:
    1. [FILE:name.py]...[/FILE] (orijinal)
    2. ```python # name.py ... ``` (yeni)
    3. ```name.py ... ``` (yeni)

Kod:
  # Format 1: [FILE:name.py]...[/FILE]
  pattern1 = r'\[FILE:([^\]]+)\]\n?(.*?)\[/FILE\]'
  
  # Format 2: ```python # name.py ... ```
  pattern2 = r'```python\s*#\s*([^\n]+)\n(.*?)```'
  
  # Format 3: ```name.py ... ```
  pattern3 = r'```([a-zA-Z0-9_\-\.]+\.[a-z]+)\n(.*?)```'

Test Sonucu:

  ✅ Fallback kullanımı: -80%
