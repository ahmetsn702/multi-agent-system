================================================================================
MULTI-AGENT ORCHESTRATION SYSTEM - PROJE ÖZETİ VE GELİŞTİRME DÖKÜMANI
================================================================================

Bu döküman, sıfırdan inşa edilen "Çoklu Ajan Orkestrasyon Sistemi" projesinde 
yapılan tüm mimari kurulumları, eklenen özellikleri ve entegrasyonları detaylı 
bir şekilde listelemektedir.

1. TEMEL MİMARİ VE ÇEKİRDEK (CORE) SİSTEMLER
--------------------------------------------------------------------------------
Sistem, ajanların birbirleriyle haberleşebildiği, ortak bir hafıza kullandığı ve 
görevleri asenkron olarak yürütebildiği modüler bir yapıda tasarlandı.

* core/base_agent.py (Temel Ajan Sınıfı):
  - Tüm ajanların türetildiği ana sınıftır.
  - Ajanların "Think" (Düşün) ve "Act" (Harekete Geç) döngülerini yönetir.
  - LLM'den gelen karmaşık cevapları (Claude'un gereksiz sohbet metinleri dahil) 
    hatasız ayrıştırarak saf JSON formatına çeviren "Regex JSON Parser" entegre edildi.

* core/message_bus.py (Mesaj Veriyolu):
  - Ajanların birbirleriyle konuşmasını sağlayan asenkron Pub/Sub (Yayıncı/Abone) sistemi.
  - Kritik, Yüksek, Normal gibi öncelik sırasına göre mesaj (Task/Feedback) iletimi.

* core/memory.py (Hafıza Yönetimi):
  - Kısa vadeli (Short-term) ve uzun vadeli bellek sistemleri eklendi.
  - Ajanların geçmiş görevleri ve sohbet bağlamını unutmaması sağlandı.

* core/orchestrator.py (Orkestratör/Yönetici):
  - Kullanıcıdan gelen ana hedefi (Goal) alır.
  - Planlayıcı ajanı tetikler, çıkan alt görevleri sırasına ve bağımsızlığına göre 
    paralel veya sıralı olarak ilgili ajanlara dağıtır.


2. AJANLAR (AGENTS) VE UZMANLIK ALANLARI
--------------------------------------------------------------------------------
Görevleri kusursuz şekilde yerine getirmek için 5 farklı uzman ajan oluşturuldu.

* Planner Agent (Planlayıcı Ajan):
  - Kullanıcı hedefini analiz eder.
  - Hedefi mantıklı JSON alt görevlerine (Sub-tasks) böler. Hangi ajana hangi 
    görevin gideceğini (Dependencies/Bağımlılıklar) belirler.

* Researcher Agent (Araştırmacı Ajan):
  - DuckDuckGo (DDGS) arama motoru entegrasyonu ile internete çıkar.
  - Kod yazılmadan önce gerekli kütüphaneleri, güncel dökümanları araştırır.

* Coder Agent (Yazılımcı Ajan):
  - Araştırmadan gelen verileri okur, Python, C# vb. dillerde ana kodları yazar.

* Critic Agent (Eleştirmen/Denetmen Ajan):
  - Yazılan kodu "Okunabilirlik", "Hata Yönetimi (Try-catch)", "Performans" 
    gibi kriterlere göre puanlar. Notu düşükse Coder ajana geri yollar ve düzeltmesini ister.

* Executor Agent (Yürütücü Ajan):
  - Onaylanan kodları güvenli bir ortamda (terminalde) çalıştırır ve test eder.


3. ARAÇLAR (TOOLS)
--------------------------------------------------------------------------------
Ajanların dış dünyayla iletişim kurmasını sağlayan araç eklentileri yazıldı.
* tools/web_search.py: Canlı internet araması donanımı.
* tools/code_execution.py: Dosya okuma, yazma ve shell (terminal) komutu çalıştırma donanımı.


4. KULLANICI ARAYÜZÜ (CLI DASHBOARD)
--------------------------------------------------------------------------------
Sistemin çalışmasını terminalde inanılmaz bir görsel şölene dönüştüren arayüz tasarlandı.
* "Rich" Python kütüphanesi kullanıldı.
* Canlı Ajan Durum Tablosu (Kim aktif, kim bekliyor).
* Canlı Mesaj Logları ekranı.
* Düşünce Süreçleri (Ajanların o anki iç sesleri).
* "Powered by AWS Bedrock" temalı kurumsal başlangıç afişi eklendi.
* Token (Jeton) ve tahmini Dolar ($) harcama paneli kodlandı.


5. YAPAY ZEKA ENTEGRASYONLARI VE AWS BEDROCK
--------------------------------------------------------------------------------
Proje boyunca maliyet ve performans testleri yapılarak sistem en tepe noktaya taşındı.

* OpenRouter Entegrasyonu (Eski Sistem):
  Sistem ilk olarak OpenRouter üzerinden kuruldu ve httpx ile asenkron API çağrıları test edildi.

* AWS Bedrock Boto3 Kurulumu (Yeni ve Mevcut Sistem):
  - OpenRouter iptal edilerek, kurumsal sistem olan Amazon AWS Bedrock'a geçildi.
  - Streaming (Kelime kelime canlı akış) desteği Bedrock invoke_model_with_response_stream ile baştan yazıldı.
  - AWS IAM Access Key okuyucu yapısı kuruldu.
  
* Kullanılan Modeller:
  1. Claude 4.6 Sonnet (us.anthropic.claude-sonnet-4-6): Kod yazımında, derin planlamada 
     ve sistemin zekasını maksimumda tutmak için kullanıldı. (Maliyet: Girdi $3 / Çıktı $15)
  2. Claude 3.5 Haiku (us.anthropic.claude-3-5-haiku-20241022-v1:0): Kullanıcının 
     "Ucuza ve çok hızlı test yapalım" talebi üzerine sistem tek satırla "Haiku" motoruna çekildi. 
     Böylece fatura maliyeti önceki sistemin %10'una düşürüldü. (Maliyet: Girdi $0.25 / Çıktı $1.25)


6. GİDERİLEN KRİTİK HATALAR (BUG FIXES)
--------------------------------------------------------------------------------
* Json Parse Hatası: Claude 4.6 gibi çok akıllı modellerin JSON üretirken öncesine veya 
  sonrasına sohbet metni yazmasından dolayı sistemin kilitlendiği fark edildi. "Regex" 
  ile sadece saf JSON'u çeken zırhlı bir "Parser" yazılarak sistem onarıldı.
* Streaming Hatası: AWS response formatına göre chunk (parça) çıkarma mantığı baştan yazıldı.

--------------------------------------------------------------------------------
